# Repo Conventions

## Why this is not a "normal" project

This is not an app. It is a **log**. Each leaf folder is a single solved (or attempted) CTF challenge plus its solver and original artifacts. There is no `src/`, no `tests/`, no build pipeline.

## Atomic unit: the challenge folder

A challenge folder is any leaf directory containing **at least one** of:
`DESCRIPTION.md`, `challenge.md`, or `challenge.json`.

The `make stats` target uses this rule and accepts both naming conventions below.

### Convention A — hand-authored (CPCTF style)

```text
<Event Year>/<NN-slug>/
├── DESCRIPTION.md   # original problem (verbatim, untouched)
├── WRITEUP.md       # my solution
├── solve.{py,sh,…}  # exploit / solver, runnable
└── <artifacts>      # vendored binaries, pcaps, sources, etc.
```

Generated by `scripts/new-challenge.sh`. Use this for any event you write up by hand.

### Convention B — platform-exported (CIT style)

```text
<Event Year>/<Category>/<NN-slug>/
├── challenge.json   # platform metadata (id, name, value, attachments, path)
├── challenge.md     # rendered description
├── README.md        # often a duplicate of challenge.md
├── writeup.md       # my solution (lowercase!)
├── files/           # original artifacts (downloaded by the exporter)
├── scripts/         # solver scripts
├── solution/        # alternative outputs (e.g. flag.txt)
└── other/           # misc
```

Plus a top-level `<Event>/challenge-index.json` that catalogs every challenge.

These come from a CTFd-style platform exporter. **Do not** modify exported files; treat them as upstream. Add your own `writeup.md` and put solvers in `scripts/`.

## Hard rules (both conventions)

- The description file is a **mirror** of the organizer's prompt. Never paraphrase. Keep original language (incl. Japanese) intact.
- The writeup file is **mine**. Suggested structure: TL;DR → Recon → Vuln → Exploit → Flag → Lessons → References.
- Solver scripts are self-contained (one file, deps inline via PEP 723 for Python).
- Original artifact filenames are **never renamed**. Add a `notes.md` if naming is confusing.
- Pick one convention per event and stick with it. Don't mix A and B inside the same event.

## Bootstrap (Convention A only)

| Style              | Path                                  | Command                                                   |
| ------------------ | ------------------------------------- | --------------------------------------------------------- |
| numbered, auto NN  | `Event/01-name/`                      | `make chall EVENT="…" NAME="…"`                           |
| numbered, force NN | `Event/42-name/`                      | `make chall EVENT="…" NAME="…" ID=42`                     |
| categorized, no NN | `Event/Web/name/`                     | `make chall EVENT="…" NAME="…" CATEGORY=Web`              |
| categorized, NN    | `Event/Web/42-name/`                  | `make chall EVENT="…" NAME="…" CATEGORY=Web ID=42`        |

Convention B is generated by your platform exporter, not by these scripts.

## Flags

- During an active CTF: write `[REDACTED]` in `WRITEUP.md`.
- After the event ends OR if the platform allows public writeups: paste the real flag.
- Never write a flag in `DESCRIPTION.md`. That file is the problem statement, not the solution.

## Big files

Use Git LFS (configured in `.gitattributes`):

```bash
brew install git-lfs
git lfs install   # one-time per machine
```

Anything matched by `.gitattributes` LFS rules is auto-staged through LFS on commit.

## Notes vs writeups

| Where           | What                                                      |
| --------------- | --------------------------------------------------------- |
| `<event>/<chall>/WRITEUP.md` | challenge-specific solution                  |
| `notes/cheatsheets/`         | reusable references (commands, formulas)     |
| `notes/playbooks/`           | step-by-step procedures (e.g. "TLS pcap recon") |
| `notes/payloads/`            | reusable payloads (XSS, SSTI, ROP gadgets)   |

Notes never live inside an event folder.

## Tools

`tools/` holds **reusable** scripts (decoders, fuzzers, wrappers around `pwntools`/`scapy`). One-shot solvers live with their challenge.

## What is NOT in this repo

- No CI/CD building binaries (the repo doesn't ship anything).
- No `tests/`. Solvers either produce the flag or they don't.
- No package manifest at root. Each solver declares its deps inline (PEP 723 for Python, etc.).
- No agentic AI files in remote — `AGENTS.md`, `.cursor/`, `CLAUDE.md` stay local (`.gitignore`).
